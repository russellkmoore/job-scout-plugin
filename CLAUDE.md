## What this is

A Claude Code plugin (`job-scout` v0.3.3) that runs daily job-sourcing for the user. Two skills do the operational work ‚ÄĒ `/scout-setup` (one-time) and `/scout-run` (daily) ‚ÄĒ backed by a small Python utility layer in `scripts/` for deterministic operations (schema, state pointer, tracker formatting, dedup, profile mining). Output is a daily report of A/B/C-tier listings with on-demand tailored resume + outreach packets.

**Core value:** A daily run reliably surfaces 5‚Äď15 actionable, well-matched job listings per top-connection company ‚ÄĒ without depending on fragile marketing-page scraping.

## How GSD planning works here

This project uses the **GSD** (Get Shit Done) workflow. Configuration in `.planning/config.json`:

- `mode: yolo` ‚ÄĒ auto-approve, just execute
- `granularity: standard` ‚ÄĒ 5‚Äď8 phases per milestone, 3‚Äď5 plans per phase
- `parallelization: true` ‚ÄĒ independent plans run simultaneously
- `model_profile: quality` ‚ÄĒ Opus for planning agents, Sonnet for execute/map
- All optional workflow agents enabled (research, plan_check, verifier, nyquist_validation, auto_advance)

**Workflow per phase:** `/gsd-discuss-phase N` ‚Üí `/gsd-plan-phase N` ‚Üí `/gsd-execute-phase N` ‚Üí verifier ‚Üí next phase.

**State persistence:** every command commits its artifacts immediately. If context is lost, `.planning/STATE.md` is the resume point.

## Key decisions locked (do not re-litigate without explicit user request)

| Decision | Rationale |
|----------|-----------|
| ATS providers in v0.4: Greenhouse, Lever, Ashby, SmartRecruiters, Workday | ~80% coverage of top-connection companies |
| Hybrid detection: top-30 batch via `/scout-detect`, lazy inline for the rest | Balance upfront work vs per-company effort |
| Pass 1 (ATS) first, Pass 2 (LinkedIn) dedupes against Pass 1 | ATS is canonical; LinkedIn supplemental |
| Dedupe key: company-slug + normalized-title fuzzy match (`rapidfuzz`) | Simple, fast, good enough |
| Dedup band: ‚Č•95% auto-merge, 70‚Äď95% review, <70% keep both | Avoids both under- and over-merging |
| Two-key dedup: loose (slug + first-3-tokens) + tight (slug + full title); both must agree | Defense-in-depth against title noise |
| ATS-sourced jobs get +1 tier bump only when posted ‚Č§30d ago | Stale ATS hits aren't stronger signal than fresh LinkedIn |
| Enrich-then-tier for ATS A-candidates (LinkedIn shared-connection lookup) | Restores warm-path signal lost by pure ATS results |
| Trust ATS on 0/error (no Chrome fallback) | Milestone-defining decision; honesty over coverage |
| Concurrent ATS calls with per-provider semaphore caps from `config.json` | Bound load per provider; tunable without code change |
| Delete marketing-page Chrome scraping path entirely (not flag-and-keep) | Removing fragility is part of the milestone definition |
| Schema: `MASTER_TARGETS_VERSION` bumps to 4 with additive `ats_slug_confidence` + `last_ats_hit_date` columns + tracker `source` + `ats_provider` columns | Per user choice; one migration, do it all |
| Stack: `httpx>=0.27,<0.29` (sync, thread-safe) + `rapidfuzz` + stdlib `concurrent.futures` + `threading.Semaphore` | `requests.Session` not thread-safe; threading beats async at ~30 concurrent requests |
| Two-factor detection gate: 200 + ‚Č•1 job + name fuzzy match ‚Č•85% | Prevents wrong-company false positives + wildcard catch-alls |
| Dispatcher distinguishes `OK_WITH_RESULTS` / `OK_ZERO` / `ERROR` | Without three states, "trust on zero" silently buries provider regressions |
| `runs.jsonl` carries per-(company, provider) hit counts + field-completion telemetry from Phase 2 onward | Makes trust-on-zero defensible from day one |
| `ats.concurrency_disabled: true` kill-switch in `config.json` | Sequential fallback without code change |

## Conventions to honor (from `.planning/codebase/CONVENTIONS.md`)

- **Single source of truth.** Schemas live in `scripts/schema.py`. File paths live in `skills/job-scout/references/file-contract.md`. Never duplicate. New paths added in Phase 1+ go in `file-contract.md` only.
- **Skill vs script split.** Skills (`SKILL.md`) hold control flow + LLM judgement. `scripts/` holds determinism (schemas, file I/O, dedup, formatting, parsing). Don't move LLM judgement into scripts; don't move determinism into prompts.
- **Sibling-script bootstrap.** Scripts that import from `schema.py` use the existing `SCRIPTS_DIR` / `sys.path.insert(0, SCRIPTS_DIR)` block. New `scripts/ats/*` modules follow this pattern.
- **`try / except ImportError` install hint.** Every third-party dependency surfaces an actionable install command on `ImportError` failure. **v0.4 change (CON-04):** recommend `pipx install <pkg>` or `python3 -m venv`, NOT `pip install --break-system-packages` (the old guidance breaks PEP 668 protection on Python 3.12+).
- **Plain `print()` for logging.** No `logging` module. Human-readable to stdout, errors to stderr with `ERROR:` prefix, machine-consumable JSON as the LAST `print()` of a CLI.
- **CLI subcommand dispatch via `sys.argv[1]`.** New `scripts/ats/detect.py` follows the existing `tracker_utils.py` / `state.py` shape (`detect-one`, `detect-batch` subcommands).
- **Schema migrations bump `MASTER_TARGETS_VERSION`.** The migration code lives in `validate_data.py`. v0.4 bumps to 4. Phase 1 enforces this.
- **`os.path.expanduser()` at the boundary.** Every CLI expands `~` exactly once when accepting paths.
- **All tracker writes go through `tracker_utils.py`.** Never write `JobScout_Tracker.xlsx` from any other module. Worker threads do NOT call `tracker_utils.append_rows` ‚ÄĒ serialize tracker writes to the main thread.
- **`${CLAUDE_PLUGIN_ROOT}`** resolves all internal plugin paths. Skill prompts use `python3 ${CLAUDE_PLUGIN_ROOT}/scripts/<script>.py ‚Ä¶` for every deterministic operation.
- **User data lives outside the plugin** in `~/Documents/JobSearch/` (state pointer at `~/.job-scout/state.json`) so `git pull` never touches it.

## Things to avoid (anti-features)

- **No retry-on-403/429 storms** within a `/scout-run` ‚ÄĒ correct backoff is "tomorrow's run." In-run retries blow the 5-min budget.
- **No generic ATS abstraction layer.** Per-provider modules are 100‚Äď200 lines each. A unifying base class that tries to handle 20 providers is the duplication anti-pattern from `CONCERNS.md` in disguise.
- **No marketing-page Chrome fallback.** Deleted in Phase 6, not flagged-and-kept.
- **No async refactor of `scripts/`.** At ~30 companies √ó 5 providers, threads + semaphores beat asyncio. `httpx` sync `Client` is the choice.
- **No OAuth flows.** Public read endpoints only.
- **No per-job LLM enrichment at fetch time.** Scoring stays in `scout-run/SKILL.md` (one LLM pass), not per-job.
- **No third-party "universal ATS detector" libraries.** Auditability + control over the two-factor gate matters.
- **No `requirements.txt` or formal dependency manifest.** Project convention: `ImportError` handlers print install hints (now `pipx`, per CON-04).
- **No general test suite.** v0.4 carve-out is `tests/test_migration.py` + per-provider fixture tests in `tests/fixtures/ats/<provider>/`. Not a broader suite.

## Model routing (cost discipline)

This project uses `model_profile: adaptive` ‚ÄĒ Opus is reserved for `gsd-planner` only. All other agents (executor, verifier, plan_checker, researcher, mapper) run on Sonnet or Haiku.

**Subagents (executor, etc.) MUST delegate generation work to the `cf-code-assistant` MCP server** when the task fits one of these patterns:

- Generating new code from a description (e.g., a new provider module, a new CLI script)
- Scaffolding tests for existing code
- Generating JSDoc/TSDoc/docstrings
- Inferring types from untyped code
- Mechanical transforms (rename, reformat, convert patterns)
- Fixing a specific bug given the code and error message
- Writing a commit message from a diff
- Scaffolding boilerplate (Worker, Python module skeleton)

**Critical rule:** never call `generateCode` or `generateWorkerBoilerplate` cold. Gather context first (read existing modules, fetch docs via Context7, look at sibling files), then pass that as the `context` parameter. Claude gathers, qwen3 generates.

**Keep with Claude (do NOT delegate):**
- Multi-file reasoning (cross-file refactors, plan execution coordination)
- Architecture decisions and tradeoff analysis
- GSD workflow steps (planning, verification, plan-checking)
- Anything requiring web search or Context7 lookups upfront

The executor's job is orchestration: read PLAN.md, gather context for each task, delegate generation to qwen3 where applicable, review output, integrate, commit. This keeps Sonnet token usage focused on judgement, not character-by-character code generation.

## Working in this repo

When the user invokes a `/scout-*` skill, the plugin runtime executes `skills/<skill>/SKILL.md` step-by-step. Skills load shared knowledge from `skills/job-scout/references/*.md` before doing anything else.

When **planning or executing a v0.4 phase**, prefer the GSD workflow (`/gsd-plan-phase`, `/gsd-execute-phase`) over ad-hoc edits. State persists in `.planning/STATE.md`.

For **brownfield exploration** (figuring out what something does), `.planning/codebase/` has up-to-date analysis. Re-run `/gsd-map-codebase` if it gets stale.

For **architecture/integration questions**, `.planning/research/SUMMARY.md` is authoritative for v0.4. STACK.md, ARCHITECTURE.md, FEATURES.md, PITFALLS.md are the underlying detail.
